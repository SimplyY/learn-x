# build

## 2026-06-15 Codex Build 复盘

### 总览

最近一周，Learn-X 的核心进展不是继续扩张功能，而是把“输入证据、周期判断、长期记忆、工程验证”拆成更清晰的责任层。

最重要的变化有三条：微信读书开始进入 Weekly Input；周期 Output 与 Memory 的人工确认边界进一步收紧；项目开始形成专用 Skill 加新鲜验证的工程方法。

与此同时，工作区仍有较多未提交改动。当前阶段应从“继续设计系统”切换到“验收、提交、真实使用”。

### 本周最核心事项

1. **Weekly Input 新增可审计的微信读书来源。**

   新增 `learn-x-input` Skill、采集脚本和测试，可按 ISO 周收集阅读时长、阅读进度、个人划线与想法。

   `notes.md` 进入 Process，`_raw.json` 只用于审计；API Key 不写入仓库。这是一次符合 Learn-X 边界的自动化：代码搬运证据，不替人判断价值。

2. **周期 Output、芒格洞察与 Memory 形成更明确的人工闸门。**

   已提交的流程把周、月、年 Output 继续限定为审稿区。芒格之魂在周期场景中只做洞察，不直接代写 Output，也不自动升级为道、法、术。

   2026-04 月报的人工勾选内容已进入 Q2 Memory，说明月度 Output 到 Memory 的链路已有真实样本，不再只是文档设计。

3. **Learn-X 开始建立项目级 AI Builder 质量体系。**

   新增 `learn-x-prompt-review` 与 `learn-x-verification`，分别约束 Prompt 契约、评测案例，以及完成声明前的新鲜证据。

   这比继续增加通用 Agent 更重要：它让 Codex 的产出从“看起来完成”转向“能证明完成”。

4. **Chat Pack 继续增强，但已出现功能堆叠风险。**

   本周加入“语言锋芒”增强器，并修正芒格之魂关闭后字数状态残留的问题；问题输入区和 Meta Prompt 区也在调整可用性。

   这些变化有明确用途，但增强器、规则和上下文联动正在变复杂。下一步需要代表性场景验收，而不是继续增加选项。

### 核心功能与系统变化

- **输入层：** Weekly Input 从飞书、Flomo、AI 摘要扩展到微信读书，且保留原始审计数据与人读笔记的分层。
- **处理层：** `learn-x-process` 继续只做确定性清洗、Process Pack、Output 空壳和候选抽取，不生成最终判断。
- **审稿层：** Weekly、Monthly、Yearly Output 维持 AI Chat 辅助、人类审核的定位；`input.json` 不成为默认上下文。
- **记忆层：** 只迁移已勾选或明确确认的内容；芒格洞察先进入候选观察池，不自动成为正式 Core。
- **工程层：** 输入、Prompt、Process、Verification 开始由不同 Skill 承担，职责比上周更清楚。

### 对 Learn-X 的意义

本周证明了一件更重要的事：Learn-X 的价值不在于拥有更多 Prompt、Skill 或自动化，而在于每一层都能回答“证据从哪里来、谁做判断、如何验证、何时进入长期记忆”。

从 AI Builder 视角看，系统正在从功能开发转向质量工程。从学习者视角看，阅读、行动与复盘开始进入同一证据链。从知行合一视角看，仍缺少的是持续真实使用，而不是更多架构表达。

### 后续建议

**必须做**

1. 把当前未提交改动按“输入采集、Prompt/Chat Pack、Output/Memory、用户认知内容”拆分审查和提交，避免一次提交混合事实、规则、代码与个人内容。
2. 用一个完整周验证微信读书采集：周边界、零阅读日、书籍过滤、划线时间、失败不覆盖都要有证据。
3. 为芒格之魂和语言锋芒各做 2 至 3 个代表性 Chat Pack 场景，检查上下文、字数状态、关闭恢复和输出目标。

**值得做**

1. 把 W24 作为真实样本跑通 `Input -> Process Pack -> Output -> Memory`，记录每一步人工判断，而不只检查文件存在。
2. 给每个项目 Skill 保留一条最小验收命令，让未来自动化能快速判断是代码失败、外部来源失败，还是人工输入缺失。
3. 下一次 Build 复盘增加“本周真正被使用的功能”一项，区分完成代码与产生价值。

**暂时不该做**

1. 不新增更多增强器，先验证现有增强器是否真的改善决策和表达。
2. 不把微信读书采集扩成自动总结或自动入 Memory；阅读证据仍应先进入 Input。
3. 不继续扩大 README 和根技术文档；实现细节留在所属 Skill、Input、Output 或 Prompt 文档。

### 当前最缺的东西

- 缺少一个从外部采集到 Weekly Output、再到人工确认 Memory 的完整周级验收记录。
- 缺少对 Chat Pack 新增强器的代表性评测，而不只是配置、语法和视觉可用。
- 缺少一次干净的版本收口，当前未提交改动会削弱后续复盘的可审计性。

### 下周优先级

1. **P0：收口版本。** 先审查并分组提交当前改动，明确哪些是正式规则，哪些仍是实验。
2. **P0：跑通真实周闭环。** 用 W24 或 W25 验证输入、Process、Output、Memory 全链路。
3. **P1：验收微信读书。** 以真实周数据验证采集正确性和不覆盖保障。
4. **P1：评测 Chat Pack。** 只测现有增强器的价值和状态恢复，不新增能力。
5. **P2：观察使用反馈。** 判断 Learn-X 是否减少重复整理，并促进真实行动。

### 收束

本周真正的进步，是 Learn-X 开始具备“证据进入、判断停留、确认迁移、完成验证”的纪律。

下一阶段不需要更大的系统。需要的是把现有系统用完一次、验证一次、提交一次，并观察它是否真的让学习更深、行动更快、维护更少。
