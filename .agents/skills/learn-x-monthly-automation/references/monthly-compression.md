# 月度事件压缩规则

月度压缩的目标不是生成月报，而是把完整原始 Input 变成可信、低重复、月份正确的 AI 上下文。原文继续保留在 `03_input/`；压缩结果只进入 `_dist`。

## 不可违反

- 只写原始来源能够支持的事实、判断和不确定性，不补造因果、人物、行动、结果或意义。
- 只保留目标月发生的事件。来源横跨月份时，删除月外事件；无法判断日期时省略并说明原因。
- 按事件压缩，不按文件机械截断。一个事件由相同人物、问题、行动和反馈组成；同一来源可拆成多个事件，多份来源也可合并为一个持续事件。
- 先删除重复、采集元数据、口头填充、目录、模板和无变化快照，再压缩正文。
- 保留时间、人物、真实行动、现实反馈、关键判断、反例、失败、限制和下一步。漂亮表达不能替代事实。
- 月记是用户月度自我反馈，保持原文，不与周材料做语义去重。

## 重要性与体积

按 UTF-8 字节数校验：

| 级别 | 判断标准 | 输出体积 |
| --- | --- | ---: |
| `core` | 改变方向；有真实行动与反馈；跨周增强；被月记确认；重要单一事件 | 3000–5000 bytes |
| `supporting` | 有具体问题、行动、判断或结果，能支撑月度主线 | 800–1500 bytes |
| `minor` | 单次、低影响、尚无反馈，但仍值得留下的信号 | 300–500 bytes |
| 省略 | 空内容、技术元数据、月外事件、无变化重复、无法验证 | 0 bytes，并写明原因 |

重要性决定事件最终大小，不使用固定压缩倍率。多事件来源的总长度等于各事件预算之和。单一重要事件即使原文很长，也只保留 3–5 KB；例如播客逐字稿应保留命题、关键论证、分歧、例子、限制与行动含义，删除寒暄、复述、口误和逐句展开。

## 来源处理

- `ai.md`：跨周合并重复母题，重点呈现判断如何变化、哪些进入行动、哪些只是工具或概念噪声；通常压缩 5–10 倍。
- `research.md` / 播客 / 访谈：先判断包含几个事件。单一核心事件输出 3–5 KB；多个事件分别评级。逐字稿不得全文进入 Process Pack。
- `weread.md`：保留各周有效阅读统计、月末进度、当月新增且去重后的个人划线与想法。相同划线只留一次；没有划线、想法或有效时长的周最多形成一个 `minor` 事件，也可以省略。
- 10 KB 按“目标月内同一类型的全部材料合计”计算，不按单文件计算。例如四至五份周度 `build.md` 必须先合并视为一个 Build 类型，再判断总量。
- 任一类型合计超过 10 KB 时，必须由 Codex 做价值审查；10 KB 是触发审查线，不是机械上限。高价值、高信噪比、几乎没有冗余的类型可以保留原文；中低价值、跨周重复、过程展开过多的部分必须跨文件合并为事件压缩。
- `daily`、`flomo` 先做确定性合并与去重；若超过 10 KB，只有在真实行动、反馈或原始自我观察占比高时才能保留全文，并在 `input.json.typeReviews` 写明理由。不能仅因它们属于固定类型而永久豁免。
- 超阈值类型中的单份高质量材料仍可使用 `passthroughs` 保留，但其余中低价值或重复材料必须压缩；所有决定均受 100 KB 总预算约束。

## 执行步骤

1. 读取 `_dist/monthly/YYYY-MM/compression-requests.json`。
2. 按请求中的路径读取每份当前原文，核对 SHA-256；不要使用旧摘要替代。
3. 先按 `source` 类型汇总月度总量；超过 10 KB 的类型整体审查。再列事件、合并同类型跨周重复，最后评级和压缩。先做事实清单，再写压缩正文。
4. 写入 `_dist/monthly/YYYY-MM/compressed-events.json`：

   ```json
   {
     "schemaVersion": 1,
     "month": "YYYY-MM",
     "events": [
       {
         "id": "E001",
         "title": "事件标题",
         "category": "action",
         "source": "research/podcast",
         "importance": "core",
         "dateRange": { "start": "YYYY-MM-DD", "end": "YYYY-MM-DD" },
         "sourcePaths": ["03_input/..."],
         "sourceHashes": { "03_input/...": "sha256" },
         "text": "压缩正文"
       }
     ],
     "passthroughs": [
       {
         "id": "P001",
         "title": "高价值原文标题",
         "sourcePath": "03_input/...",
         "sourceHash": "sha256",
         "dateRange": { "start": "YYYY-MM-DD", "end": "YYYY-MM-DD" },
         "reason": "为什么该来源高价值、高信噪比且不应压缩"
       }
     ],
     "omissions": [
       {
         "sourcePath": "03_input/...",
         "sourceHash": "sha256",
         "reason": "可核查的省略原因"
       }
     ]
   }
   ```

5. 运行 `npm run process:monthly -- --month YYYY-MM`。校验失败时修正压缩文件，不绕过哈希、日期、来源覆盖、事件体积或 100 KB 总预算。
6. 向用户报告每个类型及每个来源的原始字符数、压缩字符数、压缩率、事件数量、保留理由和省略原因。明确说明压缩由 Codex 完成，脚本只做确定性校验。

## 压缩前自检

- 如果删除这段，是否会改变对事件发生了什么的理解？
- 是否保留了行动和现实反馈，而不只剩观点？
- 是否把多个不同事件误合并成一条漂亮结论？
- 是否重复放大了跨周不变的同一内容？
- 是否存在来源中没有的因果、价值判断或确定性？
- 是否让单一长事件挤占了整个月的注意力？
